# Repository audit — 2026-07-02

Audit of `Reapper-Stack/digimon-terminus-releases`, the distribution channel for
the Digimon Terminus client. Everything a player's launcher downloads and runs
is governed by this repo, so it is audited as supply-chain-critical
infrastructure: manifest integrity, release-asset integrity, and the controls
around who/what can change them.

**Scope**: this repository only (repo tree, releases, branches, workflows,
collaborators, PRs). Other repositories on the account were not reachable from
this session and are not covered.

## Method

- Every entry in `manifest.json` (421 `files` + 5 `patches`) was cross-checked
  against the GitHub Releases API: the manifest's `sha256` and `size` were
  compared to the sha256 **digest and size GitHub records for the actual
  release asset**, for every referenced tag (v0.2.1, v0.2.2, v0.2.4, v0.2.8,
  v0.5.0, v0.5.5, v0.5.6, v0.6.6).
- Liveness of all 426 unique asset URLs verified with ranged GET requests
  (GitHub's release CDN rejects HEAD with 401 — relevant for anyone probing
  these URLs).
- Manifest structure checked for path traversal, duplicate paths, foreign
  (non-repo) URLs, malformed hashes, and overlapping patch byte-ranges
  (via `scripts/validate_manifest.py`).
- Repo configuration reviewed: branches, branch protection, collaborators,
  workflows, open PRs, release mutability.
- All 10 tracked files pattern-scanned for credentials/secrets.

## Results — integrity (clean)

- **420 of 421 `files` entries verified byte-authentic**: manifest `sha256` and
  `size` match GitHub's recorded digest for the corresponding release asset.
  Zero mismatches. The one exception is the known v0.2.1 entry (below).
- **All 5 `patches` entries** point at live assets (4× v0.2.8, 1× v0.5.5).
- **No path traversal** (`..`, absolute, drive-letter), **no duplicate paths**,
  **no non-repo URLs**, no malformed hashes, no overlapping patch ranges.
- **No secrets** found in any tracked file.
- Commit history is consistent: release-pipeline auto-commits record the shipped
  `GDMO.exe`/launcher hashes in the commit message, and those match the manifest
  at each tag inspected.

### Known & accepted

- The `DigimonTerminus.Launcher.exe.config` entry points at deleted release
  v0.2.1 (URL 404s). Accepted as-is (existing installs already carry the file).
  The URL is listed in `scripts/url-check-ignore.txt` so the URL check stays
  green; remove that line if the file is ever re-hosted.

## Results — risks and recommendations

Ordered by importance. Items marked **(settings)** can only be done in the
GitHub UI by the repo owner — none are changeable via this repo's contents.

### 1. Releases are mutable — HIGH (settings)

Release assets can currently be edited, replaced, or deleted after publishing
(`immutable: false` on every release). The manifest pins URLs into old releases
(v0.2.2 through v0.6.6 are all load-bearing), so:

- Deleting an old release silently breaks fresh installs / file repairs — this
  has already happened once (v0.2.1).
- Replacing an asset in place would make players' launchers download bytes that
  no longer match the manifest hash — or worse, if the manifest is regenerated
  to match, players get different bytes at the same URL with no history.

**Do**: enable **immutable releases** (repo Settings → General → Releases) so a
published release's tag and assets can never be altered. The new scheduled
`Check manifest URLs` workflow (below) is the detective control for deletions;
immutability is the preventive one.

### 2. Single account is the whole trust root — HIGH (settings)

One admin account (`Reapper-Stack`) controls the repo, the releases, and the
pipeline that auto-commits `manifest.json` to `main`. Anyone who takes over the
account can ship a malicious `GDMO.exe` to every player, and the launcher-side
hash check does not help — the attacker writes the matching hash into the
manifest.

**Do now**:
- Ensure **2FA** (ideally passkey/hardware-key) on the GitHub account.
- If the pipeline pushes with a token, use a **fine-grained PAT scoped to this
  repo only** with the minimum permissions (contents: write), and rotate it.
- Enable **secret scanning + push protection** (free for public repos;
  Settings → Code security). The API confirms it is not currently enabled.

**Do next (roadmap)**: adopt the detached **Ed25519 manifest signature** design
already written up in `docs/design/manifest-metadata-and-integrity.md` — an
offline private key signs `manifest.json`; the launcher embeds the public key
and refuses unsigned manifests. That is the only control here that survives a
full account compromise, because the payload players run is an executable.

### 3. Protect `main` against mistakes — MEDIUM (settings)

`main` has no protection. Because the release pipeline pushes directly to
`main`, do **not** require PRs (it would break releases — see CONTRIBUTING.md).
But a ruleset that **blocks force-pushes and branch deletion** on `main` costs
nothing and prevents history rewrites of the manifest players fetch from
`raw.githubusercontent.com/...main/manifest.json`.

Also run `scripts/validate_manifest.py` **inside the release pipeline before it
commits** — CI on `main` only reports failures after a bad manifest is already
live (CONTRIBUTING.md documents this).

### 4. CI guards — merged/added in this audit

- **PR #1 reviewed and merged** (validator + schema + `Validate manifest`
  workflow + docs). The validator was run against the live manifest: passes
  (421 files, 5 patches). It enforces structure offline but cannot detect
  deleted releases.
- **Added `Check manifest URLs`** (`.github/workflows/check-manifest-urls.yml`
  + `scripts/check_manifest_urls.py`): daily scheduled probe of every manifest
  URL (ranged GET, size cross-check against `Content-Range`). This catches the
  release-deletion failure mode within a day instead of via player reports.
  Verified against the live manifest: all 425 checked URLs OK, 1 ignored.

### 5. Housekeeping — LOW

- **GitHub Pages is enabled** (`pages-build-deployment` workflow) but the repo
  has no site content. If nothing consumes the Pages URL, disable Pages
  (Settings → Pages) to shrink surface area; if the launcher fetches the
  manifest via Pages, document that in the README instead.
- `manifest.json` ships with a UTF-8 BOM. Intentional (.NET pipeline default)
  and now documented; any new tooling must read it with `utf-8-sig`.
- Release notes are exemplary — keep embedding the shipped binary hashes in the
  auto-commit messages; it gave this audit a verifiable paper trail.

## Suggested order of adoption

1. Enable immutable releases + secret scanning/push protection + 2FA check
   *(minutes, settings only)*.
2. Add the force-push/deletion ruleset on `main` *(minutes)*.
3. Move validator into the release pipeline pre-commit step *(small)*.
4. Ed25519 manifest signing per the design doc *(the big one; needs launcher
   work — sequence: launcher verify support → publish `manifest.json.sig` → CI
   signature check)*.
